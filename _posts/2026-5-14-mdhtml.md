> 对认为HTML可以替代md的人问一句：Python能替代C++吗？

如同开篇导语的这个问题，只要是跟计算机沾点边的人都会毅然决然的回答**不能**，然而呢在近期，有个叫Thariq的博主认为“在 AI 编程工具加持下，HTML 比 Markdown 更自然”，还认定**HTML将会取代Markdown**。

这里我就只问一个问题：
- Thariq认为AI加持下HTML更自然，这就像说因为有了高级编译器和AI，直接用机器码编程更自然——能直接运行、指令可控。**那么程序员写程序为什么不写0和1？**

因为人类思维与机器思维不同。Markdown的简洁恰恰降低了人机交互的认知负荷，即便AI代劳，人依然要阅读、修改和确认，Markdown的可读性优势无可替代。当然我也并不是想说谁替代谁，而是**二者定位就不是一致的**

- HTML追求的是对渲染结果的精确控制，适合需要复杂结构和样式的场景；
- Markdown追求的是书写和阅读时的极简心智负担，适合一切以内容为中心的创作。

你可以把HTML理解成一种面向机器渲染的标记指令集，和XML同根同源；

而Markdown是面向人类思维流的一种书写约定，它要解决的根本不是怎么排版，而是怎么让人在写作时分心最少。

二者不是同类替代品，从设计哲学上就走向了两个方向。

这一点从它们各自诞生的语境里就能看得非常清楚。HTML的前身是Tim Berners-Lee在1991年提出的那套标记方案，它的目标从一开始就非常具体：让科研人员在浏览器里看到结构化的文档。所以HTML天然带着我要控制最终呈现的基因，

> 标题一定得是标题  
> 链接一定要能点  
> 图片一定要能显示  
> ——正在用石头打美国卫星的将军

它回答的问题是：浏览器，你把这段内容渲染成这样。

而Markdown是John Gruber在2004年创造出来的，他的出发点恰恰是对 HTML 写作体验的不满——写个链接要敲 `<a href="...">xxx</a>`，写个列表要手打 `<ul>` 和 `<li>`，这些标记在人类阅读源码的时候全是视觉噪音。所以 Markdown 做的事情本质上是一套“语法糖封装”：用最接近人类自然书写习惯的符号，比如用 `#` 表示标题层级、用 `-` 表示列表项、用 `**` 表示加粗，然后通过转换器把这些符号编译成对应的 HTML 标签。这决定了 Markdown 从根上就不是一套排版语言，而是一套写作速记系统。

这个起源差异直接导致了它们在日常使用中完全不同的心智模型。当你打开一个 `.html` 文件开始写内容的时候，你同时在做两件事：组织你想说的话，以及告诉浏览器怎么显示它们。你的注意力被切成两块，一块在内容本身，另一块在标签是否正确闭合、属性是否写对、嵌套有没有搞错。我前面那个故意把`<h1>`写成 `<h>` 的例子就很典型——内容的意图完全没有问题，但排版指令出了差错，浏览器就给你一个完全不渲染的结果。而在Markdown里，你打一个`# 标题`，就算你多打了一个`#` 变成 `## 标题`，内容依然清晰可见，读者根本不会因为标记符号的微小偏差而完全丢失信息。这背后反映的是两种格式对容错性的截然不同态度：

 - HTML 作为机器指令集，错误就是错误，标签不闭合可能导致整段内容塌掉；
- Markdown 作为面向人的约定，符号本身就是视觉提示，即使不经过渲染，纯文本阅读也不受影响，而大部分解析器对大多数非关键性错误也有足够的宽容度——比如你写 `#标题` 漏掉了空格，很多解析器依然能正确识别为一级标题。

再往实际工作流里看，这个差异会被放得更大。假设你要维护一份长文档，比如技术手册、项目文档或者个人知识库，里面有三五十个章节、上百个锚点链接、表格、代码块。在 HTML 里，每一次修改都意味着你要在满屏的 `<div>`、`<section>`、`<article>`、`<code>` 和 `<pre>` 标签里找到你要改的那段实际内容。你调试一个表格，得同时关注 `<table>`、`<thead>`、`<tbody>`、`<tr>`、`<td>`、`<th>` 的对应关系有没有写错；你加一个内部跳转链接，得手动写 `<a href="#anchor-id">`，还得确保对应的 `id` 属性在另一个标签上正确放置。这些心智开销跟你的写作内容毫无关系，全在跟机器指令搏斗。而在Markdown里，表格就是几行竖线分隔的纯文本，链接就是`[文字](url)`，内部跳转直接用`[文字](#标题)`，你看到的东西和你写的东西是同一套符号体系，不需要在内容记忆和标签结构记忆之间来回切换。

还有一点很容易被忽略的是版本管理的友好程度。现在大量的写作场景是和Git绑在一起的——技术文档、开源项目说明、内部知识库，都要通过diff来追踪修改。Markdown在这个场景下的优势是碾压性的。一段Markdown文本的diff，修改的就是你实际改动的那些文字和符号，增删一个段落，diff直接就告诉你哪几行变了。而HTML的diff经常是一场噩梦：你只是改了一个列表项的措辞，但因为之前有人顺手格式化了代码，整个`<ul>`区块的缩进全变了，diff显示出来红红绿绿一大片，你根本找不到你真正改的那几个字。这不是什么边缘场景，任何一个团队协作维护过HTML文档的人，都体会过被“格式化污染diff”支配的恐惧。这也是为什么GitHub、Gitee、GitLab这些协作和知识管理平台不约而同地选择Markdown作为核心书写格式，而不是让用户直接写HTML的答案。

Thariq 的说法还有一个隐含的预设，就是“AI 时代人不需要直接看代码了，所以可读性不重要”。这个预设忽略了一个关键事实：AI 写出来的内容，人还是要审核的。审核难度这一块我不赘述，就像[@酱紫君](https://www.zhihu.com/people/a74574db96f2d0e717fc410bf90dd150)说的一样，像是`<span>` 嵌套`<ul>`、 `<div>`里塞了个裸`<li>`却没有父级`<ul>`、 `<div>`当`<a>`的子孙、套上7891层`<div>`，每一层都带着行内样式`style="padding:11px;margin-bottom:45px;font-size:14px"`、那都是家常便饭，甚至——AI还怕哪个神人浏览器连粗体文本都不支持，于是给你的文案包了三层标签——`<strong><b><span style="font-weight:bold">文字</span></b></strong>` （谁家浏览器还不如IE6？）层级深到像是在搭乐高，而真正有价值的信息却被淹没在这些标签里。

换成Markdown呢？AI生成错了，你一眼就看得出来，你审核的不是排版指令，而是内容本身。AI在Markdown里也不是不犯错，但它犯的错是内容错，不是结构腐坏，你改起来也就是动几个字符的事，而不是在一堆尖括号里小心翼翼地做手术。所以AI的介入非但没有消灭可读性的需求，反而把可读性的重要性推到了前所未有的高度：因为以前代码是自己一行一行写的，心里有数；现在代码是AI一口气吐出来的，每一行都得用人眼重新验一遍，这时候如果面对的还是一堆`<span>`嵌套`<ul>`之类的天才写法，你审得越认真，血压飙得越猛,**AI生成HTML到底干过什么离谱事，用过的人心里都清楚**。
